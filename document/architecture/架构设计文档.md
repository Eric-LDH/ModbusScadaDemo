# 架构设计文档

> **版本**: v1.0 | **日期**: 2026-07-16 | **适用范围**: ModbusScada 温度监控系统

---

## 1. 系统概述

### 1.1 系统定位

ModbusScada 是一款基于 Windows Forms 的桌面 SCADA（Supervisory Control and Data Acquisition）上位机软件，通过 Modbus RTU 协议与下位机（PLC / 温度采集模块）通信，实现温度数据的实时采集、监控、报警和远程控制。

### 1.2 核心目标

- 实时采集并显示温度数据（1Hz 采样率）
- 提供可视化温度趋势曲线（5 分钟滚动窗口）
- 实现可靠的温度超限报警（死区防抖）
- 支持手动控制风扇/继电器
- 数据本地持久化 + Web 推送

## 3. 设计模式

### 3.1 事件驱动模式

服务层通过 C# 事件 `event Action<T>` 向上层通知状态变更。UI 层订阅事件后，通过 `InvokeRequired` + `BeginInvoke` 封送到 UI 线程。

```
ModbusDriver.TemperatureUpdated  →  Form1.OnTemperatureUpdated()  →  UI 更新
                                →  DataPushService.OnTemperatureUpdated()  →  HTTP 推送

AlarmManager.AlarmStateChanged  →  Form1.OnAlarmChanged()  →  UI 更新
                                →  DataPushService.OnAlarmChanged()  →  状态缓存

SerialPortManager.ConnectionChanged  →  Form1.OnConnectionChanged()  →  UI 更新
                                    →  DataPushService.OnConnectionChanged()  →  状态缓存
```

### 3.2 观察者模式

`DataPushService` 同时订阅三个事件源（ModbusDriver、AlarmManager、SerialPortManager），聚合各组件状态后统一推送。

### 3.3 锁保护临界区

| 锁对象 | 保护范围 | 用途 |
|--------|----------|------|
| `SerialPortManager.SyncLock` | 串口读写操作 | 防止轮询线程和 UI 写线圈操作竞争 Modbus 总线 |
| `ModbusDriver._isPolling` (Interlocked) | 轮询回调 | 防止定时器回调重入 |

## 4. 线程模型

```
┌─────────────────────────────────────────────────────────┐
│  UI 线程（主线程）                                        │
│  - WinForms 控件渲染与更新                                │
│  - Form1_Load 初始化                                     │
│  - 按钮点击事件（连接、开风扇、关风扇）                      │
│  - System.Windows.Forms.Timer（自动重连，3 秒间隔）         │
│  - 跨线程回调通过 BeginInvoke 封送                        │
├─────────────────────────────────────────────────────────┤
│  线程池线程 #1（System.Threading.Timer 回调）              │
│  - ModbusDriver.PollingCallback（每 1000ms）              │
│  - 读取保持寄存器、数据校验、事件发布                        │
│  - Interlocked.CompareExchange 防重入                    │
├─────────────────────────────────────────────────────────┤
│  线程池线程 #2（Task.Run 异步）                            │
│  - ModbusDriver.WriteCoilAsync（按需触发）                │
│  - 写单个线圈到从站                                       │
├─────────────────────────────────────────────────────────┤
│  线程池线程 #N（async void fire-and-forget）              │
│  - DataPushService.SendDataAsync（每次温度更新触发）       │
│  - HMAC 签名 + HTTP POST                                 │
└─────────────────────────────────────────────────────────┘
```

### 线程安全机制

- **UI 跨线程**：所有 UI 更新方法内部检查 `InvokeRequired`，必要时 `BeginInvoke`
- **轮询防重入**：`Interlocked.CompareExchange(ref _isPolling, 1, 0) != 0` 直接返回
- **串口临界区**：`lock(_serialMgr.SyncLock)` 保护所有 Modbus 总线操作
- **关闭防护**：`_isClosing`（volatile）防止窗体关闭时重连和重复清理

## 5. 核心流程

### 5.1 启动流程

```
Program.Main()
  → Application.Run(new Form1())
    → Form1_Load()
      1. LogConfig.Initialize()          # NLog 配置（文件日志，按日归档）
      2. DataLogger.Initialize()         # SQLite 建库建表
      3. AlarmManager 实例化             # 默认阈值 50℃，订阅报警事件
      4. SerialPortManager 实例化        # 创建重连定时器（WinForms Timer）
      5. ModbusDriver 实例化             # 创建轮询定时器（Threading Timer）
      6. DataPushService 实例化          # 订阅三个服务事件
      7. PlotManager 实例化              # 延迟到 formsPlot1.Visible 时初始化
      8. ScanPorts()                     # 扫描可用 COM 口
```

### 5.2 连接流程

```
用户点击"连接" → BtnConnect_Click()
  → SerialPortManager.Connect(portName)
    1. lock(_lockObj) 锁保护
    2. CleanupInternal() 清理旧连接
    3. new SerialPort(portName, 9600, None, 8, One)
    4. _serialPort.Open()
    5. ModbusSerialMaster.CreateRtu(_serialPort)
    6. FireConnectionChanged(true)
  → ModbusDriver.StartPolling()
    → _pollingTimer.Change(1000, 1000)  # 启动 1 秒间隔轮询
```

### 5.3 轮询数据流（核心循环）

```
[每 1000ms] PollingCallback() [线程池线程]
  1. Interlocked.CompareExchange 防重入检查
  2. 检查 IsConnected 和 _isDisposed 状态
  3. lock(_serialMgr.SyncLock) 进入串口临界区
  4. master.ReadHoldingRegisters(1, 0, 2)  # 从站1，地址0起2个寄存器
  5. 数据校验:
     - 范围校验: rawTemp ∈ [-40, 125] ℃
     - 跳变检测: |rawTemp - lastValidTemp| ≤ 30 ℃
     - 不通过 → sensorFault=true, validTemp=null
  6. 发布 TemperatureUpdated 事件
     → Form1.OnTemperatureUpdated() [跨线程封送 → UI 线程]
       - UpdateTemperatureUI(temp, fault)
       - UpdateRelayUI(relayOn)
       - AlarmManager.Check(temp) → 死区判断
       - PlotManager.UpdateData(temp, threshold)
       - 每 5 次: DataLogger.SaveRecord()
     → DataPushService.OnTemperatureUpdated()
       - 组装 JSON payload → HMAC-SHA256 签名 → HTTP POST
```

### 5.4 报警死区逻辑

```
AlarmManager.Check(float temp):
  overThreshold = temp > _alarmThreshold

  if overThreshold:
    _alarmCount = min(_alarmCount + 1, 4)     // 向上累加
  else:
    _alarmCount = max(_alarmCount - 1, -4)    // 向下累减

  // 触发：连续 3 次超阈值且未处于报警状态
  if !_alarmActive && _alarmCount >= 3:
    _alarmActive = true
    _alarmCount = 0        // 重置计数器
    FireAlarmChanged(true)

  // 解除：连续 3 次低于阈值且处于报警状态
  if _alarmActive && _alarmCount <= -3:
    _alarmActive = false
    _alarmCount = 0        // 重置计数器
    FireAlarmChanged(false)
```

### 5.5 资源释放流程

```
Form1_FormClosing() 或 Dispose(disposing=true)
  → CleanupResources()
    1. _isClosing = true                    # 防重入
    2. _modbusDriver.StopPolling()          # 停止轮询定时器
    3. _serialMgr.SetClosing()              # 停止重连定时器
    4. _serialMgr.Disconnect()              # 关闭串口 + 释放 ModbusMaster
    5. _modbusDriver.Dispose()              # 释放轮询定时器
    6. _serialMgr.Dispose()                 # 释放重连定时器
    7. _dataLogger.Dispose()                # 关闭 SQLite 连接
    8. _dataPushService.Dispose()           # 释放 HttpClient
    9. LogConfig.Shutdown()                 # 关闭 NLog
```

## 6. 数据模型

### 6.1 SQLite 表结构

```sql
CREATE TABLE IF NOT EXISTS temperature_log (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp   DATETIME DEFAULT CURRENT_TIMESTAMP,
    temperature REAL,
    relay_state INTEGER,
    alarm_state INTEGER
);
```

写入频率：每 5 次轮询（约 5 秒）一条记录。

### 6.2 Modbus 寄存器映射

| 寄存器地址 | 类型 | 说明 |
|-----------|------|------|
| 保持寄存器 0 | UInt16 | 温度值 × 100（如 2550 = 25.50℃） |
| 保持寄存器 1 | UInt16 | 继电器状态（0=OFF, 1=ON） |
| 线圈 0 | Bool | 风扇控制（写操作，0=关, 1=开） |

### 6.3 事件参数类

| 类 | 属性 | 说明 |
|----|------|------|
| `TemperatureEventArgs` | Temperature, SensorFault, RelayOn | 温度读数事件 |
| `AlarmEventArgs` | Active, CurrentTemp, Threshold | 报警状态变更事件 |
| `ConnectionEventArgs` | Connected, PortName | 连接状态变更事件 |

### 6.4 Web 推送 Payload 结构

```json
{
    "temperature": 25.5,
    "sensorFault": false,
    "relayOn": false,
    "alarmActive": false,
    "alarmCurrentTemp": 0.0,
    "alarmThreshold": 50.0,
    "connectionConnected": true,
    "portName": "COM3",
    "timestamp": "2026-07-16T10:30:00.0000000Z"
}
```

## 7. 技术决策记录

### 7.1 为什么用 WinForms 而非 WPF？

- 工业 SCADA 场景对 UI 复杂度要求低，WinForms 更轻量、启动更快
- 团队更熟悉 WinForms，开发效率高
- 兼容老旧 Windows 环境（Windows 7+）

### 7.2 为什么用 NModbus4 而非 NModbus？

- NModbus4 是 NModbus 的维护分支，API 稳定且文档充足
- 原生支持 .NET Framework 4.x

### 7.3 为什么用 ScottPlot 而非 OxyPlot/LiveCharts？

- ScottPlot 5 对实时数据场景性能最优（SkiaSharp 硬件加速渲染）
- API 简洁，WinForms 集成原生支持
- 社区活跃，文档完善

### 7.4 为什么数据校验在驱动层而非 UI 层？

- 驱动层是数据的唯一入口，在此校验可保证所有消费者（UI、日志、推送）获得的是已验证数据
- 分离关注点：UI 层不应承担数据校验职责

## 8. 扩展性设计

### 8.1 当前扩展点

| 扩展点 | 方式 | 说明 |
|--------|------|------|
| 新增数据消费者 | 订阅 `TemperatureUpdated` 事件 | 无需修改 ModbusDriver |
| 新增报警规则 | 订阅 `AlarmStateChanged` 事件 | 无需修改 AlarmManager |
| 替换数据库 | 实现 `DataLogger` 相同接口 | SQLite → MySQL/PostgreSQL |
| 新增推送目标 | 在 `DataPushService` 中添加 HTTP 客户端 | 当前仅单一目标 |

### 8.2 计划中改进

- 多从站支持（`SlaveAddress` 列表化）
- 串口参数配置化（波特率/数据位/校验位/停止位）

- 添加 Modbus TCP 协议支持（复用 NModbus4 的 TCP Master）
- 历史数据查询 UI

## 9. 安全考量

| 风险 | 当前状态 | 建议 |
|------|----------|------|
| Modbus RTU 无认证/加密 | ⚠️ 物理串口接入即受信 | 若升级 TCP，需加 TLS + 身份认证 |
| 风扇手动控制 | ✅ 操作确认弹窗 + 审计日志 | — |
| Web 推送签名 | ✅ HMAC-SHA256 防篡改 | 定期轮换 SecretKey |
| 配置文件密钥明文 | ⚠️ App.config 中 SecretKey 明文 | 使用 DPAPI 加密敏感配置节 |
| SQL 注入 | ✅ 参数化查询 | — |

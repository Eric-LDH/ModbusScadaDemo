## 1. 开发环境搭建

### 1.1 必备工具

| 工具 | 版本要求 | 说明 |
|------|----------|------|
| Visual Studio | 2019 (16.x) 或 2022 (17.x) | 需安装 ".NET 桌面开发" 工作负载 |
| .NET Framework | 4.7.1 SDK | VS 安装时自带 |
| Git | 任意版本 | 版本管理 |

### 1.2 首次构建

```cmd
# 1. 克隆仓库
git clone <repository-url>
cd ModbusScada

# 2. 使用 VS 打开解决方案
start ModbusScada.sln

# 3. 还原 NuGet 包
#    方式一：VS 中右键解决方案 → "还原 NuGet 包"
#    方式二：命令行
nuget restore ModbusScada.sln

# 4. 构建
#    方式一：VS 中 Ctrl+Shift+B
#    方式二：命令行
build_and_run.bat
```

### 1.3 开发硬件要求

- 至少一个可用 COM 口（物理串口或 USB 转串口适配器）
- 一个 Modbus RTU 从站设备（如温度采集模块、PLC），或使用 Modbus 模拟器

### 1.4 Modbus 模拟器（无硬件时）

推荐以下工具进行无硬件开发调试：

- **ModSim** — Modbus 从站模拟器（免费）
- **Modbus Poll** — Modbus 主站调试工具
- **Virtual Serial Port Emulator** — 虚拟串口对

典型调试拓扑：

```
ModbusScada (COM1) ←→ 虚拟串口对 ←→ ModSim (COM2，模拟从站)
```

## 3. 编码规范

### 3.1 命名约定

| 元素 | 风格 | 示例 |
|------|------|------|
| 类名 | PascalCase | `ModbusDriver`, `AlarmManager` |
| 方法名 | PascalCase | `StartPolling()`, `OnTemperatureUpdated()` |
| 私有字段 | _camelCase | `_serialMgr`, `_isClosing` |
| 公共属性 | PascalCase | `IsConnected`, `Threshold` |
| 常量 | PascalCase | `PollingIntervalMs`, `MaxDataPoints` |
| 事件 | PascalCase | `TemperatureUpdated`, `ConnectionChanged` |
| 局部变量 | camelCase | `rawTemp`, `sensorFault` |

### 3.2 注释规范

- 所有 `public` / `internal` 方法必须有 XML 摘要注释（`<summary>`）
- 复杂逻辑（如报警死区）应有行内注释解释意图
- 使用 `#region` 对 Form1.cs 进行逻辑分组

### 3.3 异常处理

```csharp
// ✅ 正确：捕获具体异常，记录日志
try
{
    _serialPort.Open();
}
catch (Exception ex)
{
    _logger.Error(ex, "串口连接失败");
    return ex.Message;
}

// ❌ 错误：吞掉异常不记录
try { DoSomething(); } catch { }
```

### 3.4 线程安全

```csharp
// ✅ 正确：UI 更新前检查线程
private void UpdateTemperatureUI(float? temp, bool sensorFault)
{
    if (InvokeRequired)
    {
        BeginInvoke(new Action<float?, bool>(UpdateTemperatureUI), temp, sensorFault);
        return;
    }
    // ... 更新控件
}

// ✅ 正确：防重入
if (Interlocked.CompareExchange(ref _isPolling, 1, 0) != 0)
    return;
```

## 4. 调试指南

### 4.1 日志查看

程序运行后，日志文件位于可执行文件同目录：

```
bin\Debug\scada.log        # 当日日志
bin\Debug\scada.20260715.log  # 历史归档
```

日志格式：`日期时间 | 级别 | 消息`

```log
2026-07-16 10:30:00.1234 | INFO | 程序启动
2026-07-16 10:30:01.5678 | INFO | NLog 初始化完成
2026-07-16 10:30:02.9012 | INFO | 数据库初始化完成: D:\...\scada_data.db
2026-07-16 10:30:15.3456 | INFO | 串口已连接: COM3 9600-8-N-1
2026-07-16 10:30:16.7890 | WARN | 温度报警触发! 当前温度: 52.3℃, 阈值: 50℃
```

### 4.2 常见调试场景

**场景 1：串口无法打开**

```
1. 检查日志中是否有 "串口连接失败" 错误
2. 确认 COM 口未被其他程序占用（如串口调试助手）
3. 检查设备管理器 COM 口状态
```

**场景 2：轮询无数据**

```
1. 确认状态栏显示 "● 已连接"
2. 查看日志是否有 "Modbus 轮询读取失败"
3. 用 Modbus Poll 验证从站是否正常响应
4. 检查从站地址和寄存器地址是否匹配
```

**场景 3：报警不触发**

```
1. 确认阈值设置正确（大于当前温度）
2. 死区需要连续 3 次超阈值（约 3 秒），等待足够时间
3. 查看日志确认温度值是否在变化
```

### 4.3 VS 调试技巧

- 在 `PollingCallback` 中设断点观察 Modbus 原始数据
- 在 `AlarmManager.Check()` 中设条件断点：`temp > 50`
- 使用 VS 的"线程"窗口观察各线程状态
- `_logger.Trace()` 级别日志可添加更细粒度的调试信息（需调整 LogConfig）

## 7. 构建配置

### 7.1 Debug vs Release

| 配置 | 优化 | 调试符号 | 输出路径 |
|------|------|----------|----------|
| Debug | 关闭 | 完整 PDB | `bin\Debug\` |
| Release | 开启 | 仅 PDB | `bin\Release\` |

### 7.2 NuGet 包管理

使用 `packages.config` 管理依赖，构建时自动从 NuGet 源还原。

**重要**：`System.Data.SQLite` 使用了 `Stub.System.Data.SQLite.Core.NetFramework` 存根包，实际 Native 二进制在运行时自动加载。

## 8. 代码审查检查清单

在进行代码变更前，请确认：

- [ ] 公共 API 有 XML 摘要注释
- [ ] 所有 UI 更新通过了 `InvokeRequired` 检查
- [ ] 异常被捕获并记录了日志
- [ ] 资源在 Dispose 中正确释放
- [ ] 没有引入新的 `async void`（除事件处理器外）
- [ ] 关键逻辑有对应的日志记录
- [ ] `_isClosing` / `_isDisposed` 标志被正确使用
- [ ] 串口操作在 `lock(_serialMgr.SyncLock)` 内执行
